# Company Parser — Design Spec

**Date:** 2026-07-01
**Status:** Approved (design), pending implementation plan

## 1. Goal

Web app to find companies by region and filters, sourced from OpenStreetMap, with
map-based region selection and export to JSON / CSV / Excel.

Primary use case: lead generation. Find businesses matching gaps — e.g. **no website**,
**no social media**, **no phone** — inside a chosen country or region.

## 2. Scope (MVP)

In scope:
- Map with clickable administrative boundaries (country / region) to select a search area.
- Filters: no website, no socials, no phone, by OSM category.
- Run search against Overpass API, map OSM tags → company records.
- Persist results + search history in Postgres (PostGIS).
- Cross-run deduplication by OSM identity.
- Export a search's results to JSON, CSV, or Excel (xlsx).
- Search history view.

Out of scope (future):
- Official Places API providers (Google / 2GIS / Yandex). Architecture leaves room
  via the `Provider` interface, but only the Overpass provider ships in MVP.
- Background job queue / async parsing of huge regions (MVP runs synchronously with
  tiling + caching; revisit if regions time out).
- Auth / multi-tenant. Single-user/local for MVP.
- Data enrichment beyond what OSM tags carry.

## 3. Decisions (locked during brainstorming)

| Decision | Choice | Why |
|---|---|---|
| Data source | OpenStreetMap via Overpass API | Free, legal, global; carries the exact tags the filters need (`website`, `contact:*`, `phone`). |
| MVP depth | Search + filters + export + DB persistence + history + dedup | User-selected. |
| Storage | Existing Postgres + PostGIS, existing Redis (both already in Docker) | Provided by user; we connect, not provision. |
| Region selection | Click administrative boundary polygon (country / region) | Most intuitive; maps cleanly to Overpass `area` scoping. |
| Legal | OSM only. **No Google/Yandex/2GIS map scraping.** ODbL attribution required. | ToS + legal risk avoidance. |

**Open assumption (flag in plan):** the user's Postgres image must expose the **PostGIS**
extension. If it is plain `postgres`, either switch the image to `postgis/postgis` or fall
back to point-in-polygon math in Go. Verify before writing the geometry migration.

## 4. Architecture

Hexagonal. The swappable data source is the core design driver: the rest of the system
depends on a `Provider` interface, never on Overpass directly.

```
backend/
  cmd/server/main.go          composition root: config, deps, http server
  internal/
    domain/                   Company, Region, Filter — pure types, no external deps
    provider/
      provider.go             Provider interface: Search(ctx, Region, Filter) ([]Company, error)
      overpass/               Overpass QL builder + HTTP client + OSM tag mapping
    service/                  search orchestration, dedup, filter application, cache lookup
    store/                    pgx + sqlc repository: companies, searches (PostGIS ST_Within)
    export/                   json.go, csv.go, xlsx.go — all write to io.Writer (streaming)
    api/                      http handlers, request/response DTOs, middleware
    cache/                    redis wrapper (Overpass response + region polygon cache)
    config/                   env-based config
  migrations/                 golang-migrate SQL files
  Makefile

frontend/
  app/                        Next.js 15 App Router routes
  components/                 Map, FilterPanel, ResultsTable, ExportBar, HistoryList
  lib/                        api client, types (shared shape with backend DTOs)
  store/                      zustand: selected region + active filters
```

### Boundaries
- `domain` depends on nothing.
- `provider`, `store`, `export`, `cache` depend only on `domain`.
- `service` orchestrates provider + store + cache.
- `api` depends on `service`; translates HTTP ↔ domain.
- `cmd/server` wires concrete implementations together.

## 5. Tech choices

**Backend**
- Go 1.23+.
- Routing: stdlib `net/http` `ServeMux` (1.22+ method+path patterns); add `chi` only if middleware needs grow.
- DB: `pgx/v5`, queries generated by `sqlc`.
- Migrations: `golang-migrate`.
- Excel: `excelize` (use `StreamWriter` for large result sets).
- CSV/JSON: stdlib `encoding/csv`, `encoding/json`.
- Concurrency: `golang.org/x/sync/errgroup` for Overpass tile fan-out.
- Logging: stdlib `log/slog`.
- Tests: table-driven; integration tests against Postgres via `testcontainers-go` (or the user's running container in CI-less local dev).

**Frontend**
- Next.js 15, App Router, TypeScript.
- Tailwind CSS + shadcn/ui.
- Map: **MapLibre GL JS** (open source, no API token) with OSM raster/vector tiles.
- Data fetching: TanStack Query.
- Results grid: TanStack Table.
- Filter/region state: zustand.

## 6. Data flow

1. Frontend loads admin-boundary polygons (GeoJSON, low-res for countries; drill into regions on zoom). User clicks a polygon → `{ regionName, osmAreaId, bbox }`.
2. `POST /api/v1/search` with `{ region, filters }`.
3. Service:
   a. Cache key from `(region, filters)`; check Redis. Hit → return cached.
   b. Miss → build Overpass QL scoped to `area(osmAreaId)`; if region is large, split into bbox tiles and fan out with `errgroup` (bounded concurrency).
   c. Map each OSM element's tags → `Company`.
   d. Apply filters (absence checks for website/socials/phone; category whitelist).
   e. Dedup by `(osm_type, osm_id)`.
   f. Persist `companies` rows (upsert on OSM identity) + one `searches` row (query, filters, result_count, created_at).
   g. Cache result in Redis with TTL.
4. Response: `{ searchId, results[], count }`.
5. Export: `GET /api/v1/searches/{id}/export?format=json|csv|xlsx` streams the file directly from the DB rows through the matching encoder.
6. History: `GET /api/v1/searches` paginated.

## 7. API

| Method | Path | Purpose |
|---|---|---|
| POST | `/api/v1/search` | Run search, persist, return results + searchId |
| GET | `/api/v1/searches` | Paginated history |
| GET | `/api/v1/searches/{id}` | One search + its results |
| GET | `/api/v1/searches/{id}/export` | `?format=json\|csv\|xlsx`, streamed |
| GET | `/api/v1/regions?q=` | Region autocomplete (Nominatim proxy) |
| GET | `/api/v1/categories` | Filterable OSM category list |
| GET | `/healthz` | Liveness |

### Request: `POST /api/v1/search`
```json
{
  "region": { "name": "Bavaria", "osmAreaId": 3600002145, "bbox": [8.9, 47.2, 13.9, 50.6] },
  "filters": {
    "noWebsite": true,
    "noSocials": false,
    "noPhone": false,
    "categories": ["shop", "amenity"]
  }
}
```

### Company record (response + export columns)
```
osmType, osmId, name, category, subcategory,
website, phone, email,
instagram, facebook, vk, telegram,
addrCountry, addrCity, addrStreet, addrHousenumber, addrPostcode,
lat, lon, openingHours
```

## 8. Filters → OSM tag logic

| Filter | Rule |
|---|---|
| no website | `website` absent AND `contact:website` absent |
| no socials | all of `contact:instagram`, `contact:facebook`, `contact:vk`, `contact:telegram` absent |
| no phone | `phone` absent AND `contact:phone` absent |
| category | element matches one of whitelist keys: `shop=*`, `amenity=*`, `office=*`, `craft=*`, `tourism=*` |

Filters are applied **after** fetch (Overpass returns the candidate set scoped by area +
category; absence filters run in Go because Overpass absence queries are awkward and slow).

## 9. Database schema (PostGIS)

```
companies
  id            bigserial pk
  osm_type      text          -- node|way|relation
  osm_id        bigint
  name          text
  category      text
  subcategory   text
  tags          jsonb         -- full raw OSM tag set
  geom          geometry(Point, 4326)
  addr_*        text columns
  website, phone, email, instagram, facebook, vk, telegram  text
  created_at, updated_at  timestamptz
  unique (osm_type, osm_id)
  GIST index on geom
  GIN index on tags

searches
  id            bigserial pk
  region_name   text
  region_area_id bigint
  filters       jsonb
  result_count  int
  created_at    timestamptz

search_results  (join: which companies belonged to which search run)
  search_id     bigint fk -> searches
  company_id    bigint fk -> companies
  primary key (search_id, company_id)
```

Region filtering at query time uses `ST_Within(geom, regionPolygon)` when a polygon is
available, else the bbox + Overpass area scoping already applied at fetch.

## 10. Error handling

- Overpass timeout / 429 → retry with backoff (bounded); surface a typed `ErrUpstreamBusy`
  so the API returns `503` with a retry hint, never a silent empty result.
- Empty result is distinct from error: `200` with `count: 0`.
- Export of unknown `searchId` → `404`.
- Invalid `format` → `400`.
- All upstream/DB errors wrapped with context (`fmt.Errorf("...: %w", err)`) and logged with `slog`.
- No silent fallbacks: a partial tile-fan-out failure fails the whole search with a clear message rather than returning partial data labeled as complete.

## 11. Testing

- `domain`, filter logic, tag mapping, Overpass QL builder: pure unit tests, table-driven.
- `export`: golden-file tests per format.
- `store`: integration tests against real Postgres+PostGIS.
- `service`: tests with a fake `Provider` and in-memory/stub cache.
- Frontend: component tests for FilterPanel + ResultsTable; one e2e happy path (select region → search → export).
- TDD per phase: red → green → refactor.

## 12. Claude Code project setup

**`CLAUDE.md`** — stack summary, make targets (`dev`, `test`, `migrate`, `lint`),
architecture map, conventions (sqlc, table-driven tests, error wrapping, no silent
fallbacks), and a **legal note**: OSM data is ODbL — attribution required; Google/Yandex/2GIS
map scraping is prohibited in this project.

**Subagents** (`.claude/agents/`):
- `overpass-expert` — Overpass QL authoring + OSM tagging schema.
- `go-backend-architect` — hexagonal Go, pgx/sqlc, concurrency, performance.
- `nextjs-map-frontend` — Next.js + MapLibre + shadcn/ui.
- `export-specialist` — CSV / XLSX / JSON streaming and edge cases.

**Skills** (`.claude/skills/`):
- `overpass-query` — recipe to turn region + filter into Overpass QL.
- `add-data-provider` — steps to add a new `Provider` adapter without touching service/api.
- `db-migration` — golang-migrate create/up/down workflow.

## 13. Development plan (phases)

1. **Scaffold + Claude config** — repo layout, `CLAUDE.md`, 4 agents, 3 skills, Makefile, docker wiring to the user's existing Postgres + Redis, `.env.example`.
2. **Backend core** — `domain` types; Overpass provider (QL builder + HTTP client + tag mapping); unit tests (TDD).
3. **Persistence** — PostGIS migrations, sqlc repo, dedup upsert, searches + search_results.
4. **API** — handlers, DTOs, validation, search orchestration + Redis cache.
5. **Export** — streaming JSON / CSV / XLSX encoders + golden tests.
6. **Frontend** — map + boundary click, filter panel, results table, export bar, history view.
7. **Integration + polish** — e2e search→export, `docker compose up`, README + ODbL attribution.

Each phase is its own TDD cycle and review checkpoint.
